# windows_local_group_member (Resource)

Manages the membership of **exactly one** member in a Windows local group on a
remote host via WinRM and PowerShell (`Microsoft.PowerShell.LocalAccounts`
module, built-in from **Windows Server 2016 / Windows 10** onwards).

Each resource instance represents one `(group, member)` pair. The
**non-authoritative** pattern means Terraform only manages the specific
membership it created, leaving out-of-band memberships undisturbed. Use
multiple `windows_local_group_member` instances — one per account — following
the `aws_iam_group` / `aws_iam_user_group_membership` split precedent.

~> **Companion resource.** `windows_local_group_member` complements
[`windows_local_group`](local_group.md), which manages the group entity itself
(name, description, SID). Do **not** mix `windows_local_group_member` instances
with a future authoritative `windows_local_group_membership` resource on the
same group.

~> **Composite resource ID.** The Terraform resource ID is
`"<group_sid>/<member_sid>"` (e.g. `S-1-5-32-544/S-1-5-21-123456789-1001`).
Both SIDs are assigned by Windows and **stable across renames** (group or
member account rename does not trigger resource replacement).

~> **BUILTIN groups fully supported.** `Administrators` (`S-1-5-32-544`),
`Remote Desktop Users` (`S-1-5-32-555`), and all other built-in groups work as
primary targets. Note that `windows_local_group` protects BUILTIN groups from
`terraform destroy`; this resource carries no such guard (it manages a
*membership*, not the group itself).

~> **All attributes are ForceNew.** There is no in-place update for a
membership: changing `group` or `member` destroys the old membership and
creates a new one.

## Example Usage

### (a) Add a domain user to the Administrators group

```terraform
terraform {
  required_providers {
    windows = {
      source  = "ecritel/windows"
      version = "~> 0.1"
    }
  }
}

provider "windows" {
  host      = var.windows_host
  username  = var.windows_username
  password  = var.windows_password
  auth_type = "ntlm"
}

# Add DOMAIN\jdoe to the built-in Administrators group.
# The group can be identified by name or SID — see example (c) for SID usage.
resource "windows_local_group_member" "admin_jdoe" {
  group  = "Administrators"
  member = "DOMAIN\\jdoe"
}
```

### (b) Add a local account by UPN to a custom group

```terraform
# windows_local_group (managed elsewhere) creates the group.
resource "windows_local_group" "app_admins" {
  name        = "AppAdmins"
  description = "Administrators of the AppSuite application stack."
}

# Add a local account using its UPN-style identity.
# member_sid, member_name, and member_principal_source are populated by Windows
# after the Add-LocalGroupMember call.
resource "windows_local_group_member" "app_admins_alice" {
  group  = windows_local_group.app_admins.name
  member = "alice@corp.example.com"
}
```

### (c) Group and member identified by SID strings

```terraform
# Both group and member can be provided as SID strings directly.
# This is the most resilient form — immune to concurrent renames.
resource "windows_local_group_member" "rdp_svc_account" {
  group  = "S-1-5-32-555"                              # Remote Desktop Users (BUILTIN)
  member = "S-1-5-21-111111-222222-333333-1002"        # svc_rdp local account
}
```

## Schema

### Required

- `group` (String) Name or SID of the target local group
  (e.g. `"Administrators"`, `"S-1-5-32-544"`, `"MyCustomGroup"`).

  Accepted formats:
  - **Group name** — resolved via `Get-LocalGroup -Name` (case-insensitive)
  - **SID string** (starts with `"S-"`) — resolved via `Get-LocalGroup -SID`

  Resolved to `group_sid` at Create time; `group_sid` is the stable handle
  for all subsequent operations (immune to concurrent group renames).

  **ForceNew:** any change destroys and recreates the resource.

- `member` (String) Identity string for the member to add.

  Accepted formats:
  - `"DOMAIN\username"` — domain user via NetBIOS domain name
  - `"MACHINE\username"` — local account with explicit machine prefix
  - `"username"` — local account (implicit local machine)
  - `"user@domain.tld"` — UPN format (domain or Azure AD account)
  - `"S-1-5-21-…-XXXX"` — direct SID string (local or domain)

  Stored **as-supplied** in state for readability; `member_sid` is the source
  of truth for drift detection and the Delete operation.

  **ForceNew:** any change destroys and recreates the resource.

### Read-Only

- `id` (String) Composite Terraform resource ID: `"<group_sid>/<member_sid>"`
  (e.g. `S-1-5-32-544/S-1-5-21-123456789-1001`). Both SIDs are stable
  across renames.

- `group_sid` (String) Security Identifier of the group, resolved at Create
  time from the `group` attribute. Stable across group renames. Forms the
  first component of the composite resource ID.

- `member_sid` (String) Security Identifier of the member as resolved by
  Windows after `Add-LocalGroupMember`. Source of truth for drift detection
  and the Delete operation (`Remove-LocalGroupMember -Member <member_sid>`).
  Forms the second component of the composite resource ID.

- `member_name` (String) Canonical display name of the member as returned by
  Windows (`Get-LocalGroupMember .Name`). Typically `"DOMAIN\username"` or
  `"MACHINE\username"`. Set to the raw SID string for orphaned AD SIDs
  (see Notes below).

- `member_principal_source` (String) Origin of the member account as reported
  by Windows. One of: `Local`, `ActiveDirectory`, `AzureAD`,
  `MicrosoftAccount`, `Unknown`. Set to `Unknown` for orphaned AD SIDs.

## Error Classification

Errors are classified into stable kinds, surfaced in the diagnostic detail
under `Kind:`:

| Kind                    | Typical cause                                                                                   |
|-------------------------|-------------------------------------------------------------------------------------------------|
| `group_not_found`       | The group resolved from `group` does not exist on the target host (EC-2, EC-5).                |
| `member_already_exists` | The resolved member SID is already in the group at Create time; use `terraform import` (EC-1). |
| `member_unresolvable`   | The `member` identity string cannot be translated to a SID on the target host (EC-3, EC-10).   |
| `member_not_found`      | The requested membership is absent after full resolution (import path only, EC-4).              |
| `permission_denied`     | The WinRM user lacks Local Administrator rights on the target host (EC-8).                     |
| `unknown`               | Catch-all for unexpected PowerShell or WinRM transport failures.                               |

## Notes

### Non-authoritative pattern

`windows_local_group_member` is **non-authoritative**: it does not enforce the
*complete* membership list of a group. Out-of-band memberships (added manually
or by other tools) are left undisturbed. If you want to enforce exact
membership, a future authoritative `windows_local_group_membership` resource
will be the appropriate tool — **do not** mix both resource types on the same
group.

### Comparison by SID, not by name

All drift detection and membership checks use **SID-based comparison**
(case-insensitive). This means:

- Renaming a member account outside Terraform does **not** cause spurious drift.
- The `member` attribute (stored as-supplied) is only used for the initial
  `Add-LocalGroupMember` call; `member_sid` governs all subsequent operations.

### Orphaned AD SID fallback (EC-6)

`Get-LocalGroupMember` has a known bug on Windows when a group contains an
orphaned Active Directory SID (account deleted from AD but still in the group
membership). In this case the resource applies a three-tier fallback strategy:

1. **Tier 1 — `Get-LocalGroupMember -SID`** (primary, PS LocalAccounts module)
2. **Tier 2 — `Win32_GroupUser` WMI query** (orphan-resilient, reads raw SIDs)
3. **Tier 3 — `net localgroup` text parsing** (last resort)
4. **All-fail** — returns an empty slice without error (conservative: no false
   positive destroy of the managed membership)

For orphaned SIDs, `member_name` is set to the raw SID string and
`member_principal_source` is set to `Unknown`.

### Do not mix with a future authoritative membership resource

If a future `windows_local_group_membership` authoritative resource is
introduced, **do not** use both on the same group. The non-authoritative
`windows_local_group_member` and any authoritative membership resource would
conflict with each other's reconciliation logic.

## Permissions

- **Local Administrator** on the target Windows host (required for
  `Add-LocalGroupMember` and `Remove-LocalGroupMember`).
- Standard authenticated WinRM user is sufficient for read-only operations.
- WinRM enabled and reachable from the Terraform runner:
  - HTTP: TCP 5985
  - HTTPS: TCP 5986
- Windows Server 2016 / Windows 10 minimum (`Microsoft.PowerShell.LocalAccounts`
  is built-in from these versions).

## Import

A `windows_local_group_member` resource can be imported using the composite
import ID `"<group>/<member>"`. Both sides accept **either a name or a SID**;
the provider auto-detects the format (values starting with `"S-"` are treated
as SIDs). The separator `/` is safe because SID strings never contain slashes.

The resource ID after import is always `"<group_sid>/<member_sid>"` regardless
of which format was used for the import ID.

```shell
# Import by group name / member NetBIOS name
terraform import windows_local_group_member.admin_jdoe 'Administrators/DOMAIN\jdoe'

# Import by group SID / member SID (most stable form)
terraform import windows_local_group_member.rdp_svc "S-1-5-32-555/S-1-5-21-111111-222222-333333-1002"

# Import by group name / member UPN
terraform import windows_local_group_member.admin_alice "Administrators/alice@corp.example.com"
```

Retrieve the member SID on the target host with:

```powershell
# Get SID of a domain user
([System.Security.Principal.NTAccount]"DOMAIN\jdoe").Translate(
  [System.Security.Principal.SecurityIdentifier]).Value

# List all members with SIDs of a group
Get-LocalGroupMember -Name "Administrators" | Select-Object Name, SID
```
