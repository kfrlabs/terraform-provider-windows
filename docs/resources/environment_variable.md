# windows_environment_variable (Resource)

Manages a single Windows environment variable (`machine` or `user` scope) on a
remote host via WinRM + PowerShell.

All registry operations use the `.NET Microsoft.Win32.Registry` API directly
(not `[System.Environment]::SetEnvironmentVariable`), which enables type-safe
control over whether the value is stored as `REG_SZ` or `REG_EXPAND_SZ` and
allows reading back raw `%VAR%` tokens for stable drift detection.

After every successful Create, Update, or Delete, the provider broadcasts
`WM_SETTINGCHANGE` via `SendMessageTimeout` so that newly started processes
(e.g. new `cmd.exe` sessions) inherit the updated environment without a reboot.
Broadcast failure is logged as a warning but does not fail the apply.

**Scope registry paths:**
- `machine` → `HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Environment` (requires Local Administrator)
- `user` → `HKCU\Environment` (no elevation required)

~> **ForceNew attributes.** Changing `name` or `scope` destroys the existing
variable and creates a new one. There is a brief window during a name rename
where neither the old nor the new variable exists; interrupted applies should be
retried.

~> **PATH management.** List-style `PATH` semantics (add/remove individual
segments) are intentionally out of scope. Use the future `windows_path_entry`
resource for that purpose.

## Example Usage

### Machine-scope plain string (REG_SZ)

```terraform
resource "windows_environment_variable" "java_home" {
  name  = "JAVA_HOME"
  value = "C:\\Program Files\\Java\\jdk-17"
  scope = "machine"
}
```

### Machine-scope expandable string (REG_EXPAND_SZ)

```terraform
resource "windows_environment_variable" "tools_dir" {
  name   = "TOOLS_DIR"
  value  = "%ProgramFiles%\\MyTools"
  scope  = "machine"
  expand = true
}
```

### User-scope variable

```terraform
resource "windows_environment_variable" "my_token" {
  name  = "MY_APP_TOKEN"
  value = "abc123"
  scope = "user"
}
```

### Empty value (valid on Windows)

```terraform
resource "windows_environment_variable" "cleared" {
  name  = "LEGACY_VAR"
  value = ""
  scope = "user"
}
```

### Using the data source for cross-resource reference

```terraform
resource "windows_environment_variable" "java_home" {
  name  = "JAVA_HOME"
  value = "C:\\jdk17"
  scope = "machine"
}

data "windows_environment_variable" "java_home_read" {
  name       = windows_environment_variable.java_home.name
  scope      = "machine"
  depends_on = [windows_environment_variable.java_home]
}

output "java_home_value" {
  value = data.windows_environment_variable.java_home_read.value
}
```

## Schema

### Required

- `name` (String) Windows environment variable name (e.g. `JAVA_HOME`). Must be
  non-empty and must not contain `=` (the Windows env-block delimiter). Case is
  preserved verbatim; Windows env lookups are case-insensitive at the OS level
  but the stored casing is kept. **ForceNew:** changing `name` destroys the old
  variable and creates a new one.
- `scope` (String) Scope of the environment variable. `"machine"` targets
  `HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Environment` and
  requires **Local Administrator** privileges. `"user"` targets
  `HKCU\Environment` (no elevation required). **ForceNew:** changing `scope`
  destroys the old variable and creates a new one.
- `value` (String) Value of the environment variable, written verbatim to the
  registry. An empty string is valid. For `expand = true`, the value should
  contain `%VAR%` tokens; they are stored as-is and read back unexpanded for
  stable drift detection. Updatable in-place (no resource replacement).

### Optional

- `expand` (Boolean) Controls the Windows registry value kind used to store the
  variable. `false` (default): stores as `REG_SZ` — plain string. `true`:
  stores as `REG_EXPAND_SZ` — Windows expands `%VAR%` tokens when the variable
  is read by applications via `ExpandEnvironmentStrings`. The provider always
  reads the raw (unexpanded) value for drift detection. Updatable in-place:
  changing `expand` rewrites the registry kind without replacing the resource.
  Default: `false`.

### Read-Only

- `id` (String) Terraform resource ID. Composite string formatted as
  `"<scope>:<name>"` (e.g. `"machine:JAVA_HOME"` or `"user:MY_APP_TOKEN"`).
  Identical to the import ID format. Set by the provider at Create time; never
  mutated by Update.

## Notes

### REG_EXPAND_SZ and drift detection

When `expand = true`, Windows stores the value as `REG_EXPAND_SZ`. Applications
calling `ExpandEnvironmentStrings` will receive the expanded form (e.g.
`C:\Program Files\MyTools`). The Terraform provider **always reads the raw
`%VAR%` tokens** (using `DoNotExpandEnvironmentNames`) so that drift detection
compares `%ProgramFiles%\MyTools` against `%ProgramFiles%\MyTools` — not against
the expanded path, which would change across machines or OS reinstalls.

### WM_SETTINGCHANGE broadcast

After every Create, Update, or Delete the provider calls `SendMessageTimeout`
(user32.dll) to broadcast `WM_SETTINGCHANGE` with `lParam = "Environment"`.
This tells running processes (Explorer, new `cmd.exe` windows) to refresh their
in-memory environment block.

In restricted or headless WinRM sessions (e.g. Windows Server Core with no
desktop session), the broadcast may time out or fail. This is non-fatal: the
provider logs a `WARN`-level message but does **not** fail the apply. The
registry write is authoritative; new processes will inherit the correct value
regardless.

### Permissions

- `scope = "user"` — no elevated privileges required; operates on `HKCU`
  scoped to the WinRM authentication identity.
- `scope = "machine"` — requires **Local Administrator** on the WinRM target
  (writes to `HKLM`). Use WinRM credentials with the appropriate role.

### Variable name case sensitivity

Windows environment variable lookups are case-insensitive at the OS level, but
the registry stores the exact casing written. The provider preserves the casing
declared in your configuration. Drift is detected only on value or kind
mismatches, not on casing.

### Import limitations (colon in name)

The import ID format `<scope>:<name>` uses the first colon as the delimiter.
A variable name containing a colon would be misinterpreted by the import parser.
This is a known v1 constraint; such names must be managed via `config + apply`
rather than `terraform import`.

## Import

A `windows_environment_variable` resource can be imported using its composite ID:

```
<scope>:<name>
```

```shell
# Import a machine-scope variable
terraform import windows_environment_variable.java_home 'machine:JAVA_HOME'

# Import a user-scope variable
terraform import windows_environment_variable.my_token 'user:MY_APP_TOKEN'
```

After import, run `terraform plan` to verify that the declared configuration
matches the live registry state. The `value` and `expand` attributes are
populated from the registry during import.
